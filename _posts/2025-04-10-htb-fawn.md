Fawn. Very Easy. Starting Point. FTP left wide open. Let's see what we're working with.

## What even is this?

Tier 0 again. HTB hands you a machine, you break in, you grab a flag. This time the lesson is FTP — File Transfer Protocol. Old, widely used, and when misconfigured? Free entry.

## Step 1 — Wait, where are you?

Spawned the machine, got the IP, went to ping it —

```bash
ping 10.129.96.106
```

Nothing. Checked if VPN was alive:

```bash
ip a | grep tun
```

> `ip a` shows your network interfaces. `grep tun` filters for the VPN tunnel — if it's there, you're connected.

VPN was fine. Machine wasn't responding. Reset it, restarted it, still nothing. Turns out I had the lab VPN config loaded — but Starting Point machines run on a completely different network. Two different VPNs. I was knocking on the wrong door the whole time.
Downloaded the correct Starting Point config, connected — machine reachable. Moving on.

## Step 2 — Knock knock, nmap

```bash
nmap -sV 10.129.96.106
```

> `-sV` finds open ports AND figures out what's actually running on them.

Port 21 open. FTP. Version — vsftpd 3.0.3. That's all we needed.

## Step 3 — Anonymous? Sure, come in.
FTP is a protocol for transferring files between machines. Port 21 is where it listens. Now here's the thing — FTP can be configured to allow anonymous login. No real credentials. Anyone gets in.

```bash
ftp -a 10.129.96.106
```

> `-a` automatically logs you in as anonymous. No username typing, no password. Just in.

And it just... let me in. No fight. Nothing.

## Step 4 — Flag goes brr

```bash
ls
get flag.txt
```

> `ls` lists files. `get` downloads them to your machine.

```bash
cat flag.txt
```

Done.

## What did I actually learn?

Anonymous FTP on a live machine is just leaving your door unlocked and hoping nobody tries it. Somebody always tries it.
Also — always check which VPN config you're on before you spend 20 minutes blaming the machine.
Fawn was easy. But that VPN thing? That'll catch you on harder machines too.
